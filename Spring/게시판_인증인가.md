# 공부 계획
0. 자료조사 
  - 관련 학습 키워드가 뭐가 있는지 조사. 반복 발견 시 학습.
  - 검색했을 때 나오는 키워드들
  - 인증(Authentication): 사용자가 본인임을 증명하는 과정. 폼 로그인(ID/PW), Oauth2 소셜로그인(kakao, google, naver), JWT 토큰 기반 인증 등.
  - 인가(Authorization): 로그인된 사용자가 특정 리소스에 접근할 권한이 있는지 확인하는 과정
  - spring security: 보안 프레임워크. 구현할 때 필요할 듯.
  - 필터 체인(컨트롤러 도달 전)
  - SecurityContext, UserDetailsService. 같은 핵심 구조(DB) 학습
  - SecurityContext와 인증객체 꺼내기: 컨트롤러에서 현재 로그인한 사용자 정보를 가져와야하는데 어떻게 가져오는지 파악. @AuthenticationPrincipal, @SecurityContextHolder. 이는 게시판에서 내가 쓴 글만 수정 가능을 구현할 떄 필수이다.
  - Access token, refresh token: 사용자(브라우저)의 정보를 담고 있고, access token은 보통 15분, refresh token은 2주라고 알고있음.
  - Oauth 2.0: 구글, 카카오, 네이버, 깃허브 같은 외부 서비스 정보를 빌려와 로그인하는 방식(Resource Owner, CLient, Authorization Server, Resource Server 역할 파악), spring-boot-starter-oauth2-clint로 구현
  - 비밀번호 BCrypt 암호화 알고리
  - 쿠키, 세션, 토큰: 인증방식 3대장 
  - Cookie (쿠키): 브라우저에 저장되는 작은 데이터 조각. 보안 옵션(`HttpOnly`, `Secure`)이 왜 필수인지 조사하기.
  - Session (세션): 서버가 사용자의 상태를 기억하는 법. `Session ID`가 전달되는 과정 이해하기.
  - JWT (JSON Web Token): 토큰 기반 인증. 왜 별도의 DB 조회 없이도 인증이 가능한지(Stateless) 구조 파악하기.
  - 회원가입, 로그인 플로우 조사
  - 사용자가 로그인 했을때 어떻게 서버가 어떻게 발급하고 등의 구체적인 동작원리 이해
  - CSRF: Cross-site Request Forgery: spring security는 기본적으로 csrf 보호를 켜놓는다. 그래서 403 만나게 된다. 왜 끄는지 왜 켜는지 파악 필요.

# 1. 동작원리 이해
  - 쿠키 vs 세션 vs 토큰 비교
  - 세션 기반 로그인 플로우 (요청 -> 필터 -> SecurityContext -> 응답)
  - BCrypt 단방향

웹개발에서 쿠키, 세션, 토큰 모두 HTTP 프로토콜의 stateless (무상태) 특성을 보완하기 위해 사용하는 기술들. (아 사용자 정보(상태)를 보안 때문에 저장하지 않는 것 때문에 귀찮음이나 문제가 생기는데 그걸 해결하는 것들이구나.)
서버가 클라이언트를 기억하기 위한 방식들이다.

1. 쿠키

쿠키는 사용자(브라우저)에 저장되는 작은 텍스트 파일.
인증의 결과로 서버가 응답 헤더에 Set-Cookie를 포함하면, 브라우저는 이를 저장했다가 다음 요청부터 자동으로 헤더에 담아보낸다.

2. 세션

사용자가 로그인하면, 그 로그인 상태를 서버의 메모리나 DB에 저장. (stateful)
클라이언트에게 ID만 쿠키에 담아 전달.
클라이언트는 이후 요청마다 ID 제시. 서버는 이를 확인

3. 토큰

인증에 필요한 정보를 암호화해서 클라이언트가 들고 있게(로컬 스토리지 or 쿠키)
서버가 유효한 사용자에게 서명된 토큰 발급.
클라이언트는 요청 헤더(Authorization)에 토큰을 담아 보냄. 서버는 토큰의 유효성만을 검사.
